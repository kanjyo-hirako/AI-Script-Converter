# AI 小说转剧本工具 — 3天 MVP 执行方案

## 项目目标

将 3 个章节以上的中文小说文本，通过 AI 自动转换为结构化 YAML 剧本。用户可在 Web 编辑器中查看、编辑、导出。

---

## 技术选型总览

| 层级 | 技术 | 说明 |
|------|------|------|
| 前端框架 | **Vue 3 + TypeScript** | Composition API，单文件组件 |
| 构建工具 | **Vite** | 快，配置简单 |
| 样式 | **Tailwind CSS** | 原子化 CSS，不用写一堆样式文件 |
| YAML 编辑器 | **Monaco Editor** | VS Code 内核，语法高亮+折叠 |
| 后端 | **Express.js + TypeScript** | 轻量 API 代理 |
| AI 调用 | **openai SDK** | 国内模型全兼容 OpenAI 格式，一个 SDK 搞定 |
| YAML 校验 | **yaml + ajv** | 解析 + Schema 校验 |
| 文件解析 | **mammoth(.docx)** | 浏览器端解析 Word 文件 |

**为什么用 openai SDK 接国内模型？**
DeepSeek、通义千问、Moonshot、智谱 全都提供 OpenAI 兼容 API。只需要切换 `baseURL` 和 `apiKey`，不需要为每个模型写适配代码。

---

## 前端框架选型理由：Vue 3

- 单文件组件（`.vue`）开发效率高，template + script + style 一个文件搞定
- Composition API + `ref`/`reactive` 响应式数据，适合处理复杂状态
- 国内生态好，中文文档齐全
- 对比 React：学习曲线更低，模板语法更直观，适合偏前端的开发者快速上手

## YAML Schema 设计

### Schema 定义

```yaml
screenplay:
  meta:
    title: string           # 剧本标题
    author: string          # 编剧（默认为用户）
    genre: string           # 类型：剧情/喜剧/悬疑/科幻等
    logline: string         # 一句话概要
    source_title: string    # 原著小说名
    created_at: string      # 生成时间 ISO 8601

  characters:
    - id: string            # 唯一ID，如 "C01"
      name: string          # 角色名
      description: string   # 角色简介
      age: string           # 年龄/年龄段
      role: string          # protagonist / antagonist / supporting / minor

  locations:
    - id: string            # 唯一ID，如 "L01"
      name: string          # 地点名
      type: string          # INT / EXT / INT/EXT
      description: string   # 地点描述

  scenes:
    - id: string            # 场景ID，如 "S01"
      chapter_ref: number   # 对应原著第几章
      heading: string       # 场景标题，如 "INT. 咖啡馆 - 夜晚"
      location_id: string   # 引用 locations 中的 id
      time_of_day: string   # 白天/夜晚/黄昏/清晨等
      summary: string       # 场景概要（一两句话）
      content:              # 场景内容数组，按顺序排列
        - type: "action"    # 动作/场景描写
          text: string
        - type: "dialogue"  # 对话
          character_id: string
          parenthetical: string  # 可选，括号注释如"（怒）""（低声）"
          text: string
        - type: "transition" # 转场
          text: string       # 如 "切至"、"淡出"、"闪回"
        - type: "note"       # 编导注释（可选）
          text: string

  # 可选：三幕结构标注
  act_structure:
    - act: number           # 1, 2, 3
      scene_ids: string[]
      summary: string
```

### 为什么这样设计

**1. characters 和 locations 独立定义，scenes 用 id 引用**

好处：
- 同一角色/地点出现多次时不用重复写
- 前端可以单独展示角色列表、地点列表
- 统计方便（有多少角色、多少场景）
- 修改角色名时只需改一处

**2. content 用 type 联合类型**

剧本只有四种元素：动作描写、对话、转场、注释。用 `type` 字段区分，前端可以做不同的渲染样式（对话居中、动作斜体、转场右对齐）。

**3. chapter_ref 保留章节映射**

方便作者回溯原著，也支持"只重新生成第3章"这种增量操作。

**4. act_structure 可选**

不是所有剧本都严格遵循三幕结构。标记为可选，不强制。

**5. YAML 而非 JSON**

- 人类可读可编辑（作者要手动改）
- 支持注释（作者可以在 YAML 里加备注）
- 缩进结构直观
- 所见即所得的编辑体验比 JSON 好

**6. id 命名规则**

- 角色：`C01`, `C02`, ...
- 地点：`L01`, `L02`, ...
- 场景：`S01`, `S02`, ...
- 简短、有序、方便引用

### 后端调用代码核心

```typescript
// server/src/services/ai.ts
import OpenAI from 'openai';

export function createAIClient(apiKey: string, baseURL: string) {
  return new OpenAI({ apiKey, baseURL });
}

export async function convertChapter(
  client: OpenAI,
  model: string,
  systemPrompt: string,
  userPrompt: string
) {
  const response = await client.chat.completions.create({
    model,
    messages: [
      { role: 'system', content: systemPrompt },
      { role: 'user', content: userPrompt },
    ],
    temperature: 0.3,        // 低温度，输出更稳定
    response_format: { type: 'json_object' },  // 要求 JSON 输出
  });
  return JSON.parse(response.choices[0].message.content);
}
```

## 长文本处理策略

不同模型上下文窗口不同，需要分块处理：

| 模型 | 上下文窗口 | 策略 |
|------|-----------|------|
| DeepSeek Chat | 64K | 单章 < 50K tokens 直接发送 |
| Qwen Plus | 128K | 单章 < 100K tokens 直接发送 |
| Moonshot 32K | 32K | 单章 < 25K tokens 直接发送 |
| Moonshot 128K | 128K | 单章 < 100K tokens 直接发送 |

**分块逻辑**（当单章过大时）：
1. 按段落（双换行 `\n\n`）切分
2. 每块约 3000-5000 字
3. 逐块转换，角色表和场景表累积传递
4. 最后合并所有块的输出

## 注意事项

1. **API Key 安全**：MVP 阶段 Key 存 localStorage 并通过前端传给后端。正式版应改为后端存储或加密传输。
2. **流式输出**：MVP 先用非流式（等完整响应），Phase 2 再加 SSE 流式显示生成进度。
3. **并发限制**：逐章串行调用，避免触发模型 API 限流。Phase 2 可加队列和并发控制。
4. **成本预估**：DeepSeek deepseek-chat 约 ¥1/百万 tokens，一个 5000 字小说转换成本约 ¥0.05-0.1。
5. **模型选择建议**：DeepSeek 性价比最高，推荐作为默认模型。Moonshot 128K 适合超长文本。
